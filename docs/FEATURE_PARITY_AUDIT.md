## Highest-Risk Gap Selected

Reservation concurrency was selected because it directly affects inventory integrity and overbooking. This loop added:

- `ReservationIdempotencyRecord`
- `Idempotency-Key` header support
- serializable reservation creation transaction
- serialization retry
- conflict response with HTTP 409
- overlap validation inside the transaction using `existing.checkIn < requested.checkOut AND existing.checkOut > requested.checkIn`

Remaining risk: a true multi-connection parallel database integration test is still needed to prove behavior under real concurrent traffic.

## Loop 7 Front Desk Operations

This loop added a dedicated Front Desk operations layer instead of relying only on dashboard cards.

- Backend: `GET /front-desk/:propertyId` with JWT, `frontdesk.view`, tenant property access, subscription entitlement `pms.frontdesk`, DTO validation, and audit logging.
- Frontend: `/front-desk` workspace linked from the main app shell.
- Data returned: arrivals, departures, in-house reservations, pending check-ins/check-outs, room readiness counts, dirty/clean/maintenance/blocked rooms, outstanding balances, new reservations, recent cancellations, no-shows, revenue counters, alerts, and quick action definitions.
- Permissions added: `frontdesk.view`, `frontdesk.manage`, `reservations.checkin`, `reservations.checkout`, `reservations.assign-room`, `reservations.no-show`, `payments.manage`.
- Tests added: FrontDeskService verifies tenant-scoped success, cross-tenant denial before data reads, and subscription entitlement denial before data reads.

Remaining risk: Loop 8 replaced redirects with action modals and backend endpoints, but production parity still requires deeper policy configuration, real PDF rendering/download validation, email delivery, channel sync outbox processing, and browser E2E workflow tests.

## Loop 8 Front Desk Domain Actions

This loop added local domain actions behind the Front Desk workspace.

- Backend endpoints:
  - `POST /reservations/:reservationId/assign-room`
  - `POST /reservations/:reservationId/change-room`
  - `POST /reservations/:reservationId/check-in`
  - `POST /reservations/:reservationId/check-out`
  - `POST /reservations/:reservationId/no-show`
  - `POST /reservations/:reservationId/payments`
  - `POST /reservations/:reservationId/invoices`
  - `POST /housekeeping/tasks`
  - `POST /maintenance/issues`
  - `POST /maintenance/rooms/status`
- Domain rules added: reusable reservation transition validator, structured conflict errors, room availability checks, overlap checks inside serializable transactions, Decimal-safe financial summaries, payment refund validation, checkout housekeeping task creation, and room status synchronization.
- Database changes: lifecycle actor/timestamp fields on Reservation, `ReservationRoomChange`, Payment type/idempotency metadata, invoice currency/PDF metadata, expanded housekeeping/maintenance statuses and operational fields.
- Frontend changes: `/front-desk` quick actions now open domain action modals instead of redirecting to generic CRUD pages, submit to the real API, show reservation/room context, display structured backend errors, and refresh overview data after success.
- Tests added: Front Desk action service tests for room assignment success, maintenance-room denial, overlap conflict, invalid check-in transition, idempotent payment behavior, and checkout housekeeping synchronization.

Remaining risk: policies for timezone windows, early check-in, deposit requirements, force checkout permission split, immutable fiscal invoice PDFs, email/WhatsApp workflow messages, and channel-sync outbox events are not fully production-complete.

## Loop 9 Live Reservation Calendar

This loop replaced the static calendar mock with a real tenant-scoped reservation timeline.

- Backend APIs:
  - `GET /calendar/timeline`
  - `PATCH /reservations/:reservationId/calendar-move`
  - `PATCH /reservations/:reservationId/calendar-resize`
  - `POST /calendar/blocks`
- Database changes: `CalendarBlock`, `CalendarBlockType`, room timeline indexes, and reservation timeline indexes.
- Frontend: `/calendar` page with property/date/view filters, room type/room/status/source/search filters, include cancelled/no-show toggles, room-by-date timeline, reservation bars, block bars, mobile list fallback, reservation details drawer, create-from-slot drawer, accessible edit-stay form, and block creation.
- Concurrency: calendar move uses serializable transaction, overlap checks, calendar block checks, terminal-state validation, maintenance-room validation, and optimistic concurrency through `expectedUpdatedAt`.
- Reservation creation from an empty slot uses the existing `POST /reservations` service and sends an `Idempotency-Key`.
- Tests added: CalendarService timeline fetch, cross-tenant denial, range validation, move/audit success, stale move rejection, overlap rejection, and block creation.

Remaining risk: visual drag/drop has a basic HTML drag path plus the accessible edit modal, but no Playwright browser E2E has verified rollback/drag behavior. Calendar blocks prevent calendar moves and show on the timeline, but they are not yet integrated with external OTA/channel outbox sync.

## Loop 9 Hardening

This hardening pass resolved several locally solvable production gaps before Loop 10:

- Added `PATCH /calendar/blocks/:blockId` and `DELETE /calendar/blocks/:blockId`.
- Calendar block updates now enforce tenant access, subscription entitlement, stale `expectedUpdatedAt`, room/property ownership, reservation conflict checks, block conflict checks, audit logging, and inventory outbox creation.
- Calendar block deletion is soft-delete based and prevents deleting unresolved maintenance-linked blocks.
- Added `InventoryOutboxEvent` with pending/processing/processed/failed states, idempotency key, tenant context, and polling indexes.
- Added local `InventoryOutboxService` dispatcher that marks pending events processed locally; real channel provider dispatch remains external activation.
- Calendar move, block create/update/delete, reservation creation, room assignment/change, check-in, check-out, no-show, and maintenance room blocking now write outbox events.
- Added `PaymentIdempotencyRecord` with request hashing and same-key/different-payload rejection.
- Added `PropertyOperationalPolicy` foundation and enforced core clean-room, guest-details, identification, deposit, checkout-payment, force-checkout, invoice, and housekeeping policy flags.
- Added real invoice PDF generation/download at `GET /invoices/:invoiceId/download` using tenant-safe local storage and a non-fiscal disclaimer.
- Ran repository Prettier formatting so `npm run format:check` now passes.

Still incomplete: Playwright browser E2E, full Loop 10 room/rate/availability engine, strict concurrent payment duplicate integration tests, richer invoice PDF layout, real cloud storage, and real OTA/channel outbox provider dispatch.
